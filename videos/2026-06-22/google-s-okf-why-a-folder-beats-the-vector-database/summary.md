## 一句話總結
AI 業界花了兩年用向量資料庫（RAG）給模型記憶，如今 Karpathy 與 Google 證明：一個會自我維護的純文字資料夾（OKF）反而做得更好、更便宜。

## 重點摘要
- 過去兩年的主流做法是 RAG：把文件切成數千個片段、轉成向量、存進專用向量資料庫，查詢時撈出最相近的片段餵給模型——能用，但每次查詢都從零開始，模型無法「記住」先前推導過的關聯。
- OpenAI 共同創辦人、前 Tesla AI 主管 Andrej Karpathy 在四月於 GitHub 提出「LLM Wiki」概念，把 RAG 反轉：知識只建構「一次」成一個彼此互連的純文字資料夾，像工程師讀程式碼那樣讓模型閱讀。
- 關鍵在於 wiki 是「AI 寫的，不是你寫的」——你提供新材料並提出好問題，AI 負責摘要、交叉引用、歸檔與維護。Karpathy 的比喻：Obsidian 是 IDE，LLM 是程式設計師，wiki 是程式碼庫。
- 6/12 Google Cloud 把這個社群點子正式化為規格「Open Knowledge Format（OKF）」：一個 bundle 就是一個資料夾，每個檔案代表一個概念，路徑即名稱，檔案間連結形成知識圖，唯一硬性規則是每個檔案要用一個欄位標明自己的類型。
- OKF 規格刻意極簡且高度寬容：要求讀取工具原諒未知欄位、壞掉的連結、甚至無法解析的檔案；但 Google 刻意拿掉了 Karpathy 原本「AI 如何維護 wiki」的指示——保留資料夾，卻省略了讓它「保持鮮活」的部分。
- 資料夾勝出的三個理由：(1) 運算時機——RAG 在查詢時思考，wiki 在建構時一次完成連結與摘要，之後只需讀取現成答案；(2) 規模——每個資料夾附帶簡短目錄，模型先讀目錄再挑出需要的那一個檔案，不會被上萬個檔案塞爆；(3) 純文字——可放進 Git、做 diff、在 PR 中審查、打包離線執行，不需資料庫、伺服器或 API key。
- 釐清兩個誤解：OKF 不與 MCP 競爭（MCP 是搬運資料的管線，OKF 是被搬運的貨物）；也不是 SEO 技巧（是給自己 agent 用的私有知識，無助於搜尋引擎找到你）。
- 四大隱憂：(1) 規格沒有任何讓內容保持更新的機制，單人維護很美好，但共享團隊資料夾一個月就會過時；(2) 語言模型其實不擅長大規模寫出乾淨 markdown，會搞砸格式、捏造不存在的連結，而 Google 的「解法」竟是改規格叫讀取端去原諒髒亂；(3) 規格只標準化「容器」不標準化「意義」，那唯一必填欄位是自由文字，你寫 BigQuery table、我寫 table、他寫 relational asset 都合法卻各說各話。
- 策略真相：OKF 出自 BigQuery 團隊而非 Google AI 實驗室——範例資料集都跑在 BigQuery，參考工具靠 Gemini，最終 bundle 最容易倒進 Google 剛改名的知識產品。發布當天 Google 以外幾乎無人使用，可能淪為 Google 墳場的下一個專案。

## 關鍵結論／takeaways
- 給模型記憶不需要昂貴的專用基礎設施——一個組織良好的純文字資料夾就能勝過向量資料庫，這個底層觀念已經贏了，無論 OKF 這個格式本身存不存活。
- 不要自己寫 wiki：讓 AI 負責摘要、交叉引用與歸檔，你只負責提供材料和提出好問題。
- 把知識當程式碼管理：純文字 + Git，可 diff、可 PR 審查、可離線打包，門檻是「只要能打開檔案就行」。
- 「一次建構、多次讀取」勝過「每次查詢重新推導」——把昂貴的連結與摘要成本前置付一次。
- OKF 不能取代 MCP，兩者互補：MCP 是管線，OKF 是貨物；也別期待它幫你做 SEO。
- 真正的護城河是看不見的：兩個外觀相同的資料夾，一個在生產環境撐得住、一個慢慢腐爛——靠的是組織方式（哪些鎖定、哪些允許 AI 改寫、如何防止長期漂移），格式本身給不了你這個能力。
- 採用前先想清楚兩件規格沒解決的事：誰來防止內容過時，以及團隊如何對「檔案類型欄位」的用語達成共識。
